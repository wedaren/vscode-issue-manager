# 个人助手（执行官）系统架构文档

> 适用版本：feat/llm-chat-and-browser-tools

---

## 一、核心理念

**个人助手就是你自己。**

传统聊天角色是独立的 AI 人格，用户与每个角色单独对话。个人助手不同——它是用户在 AI 世界里的「代理人」：

- 你对助手说的话 = 你对整个 AI 团队下达的指令
- 助手委派给「研究员」处理的任务 = 你亲自和研究员对话
- 助手看到的所有交互记录 = 你可以看到的记录

因此，助手委派产生的对话文件和你手动开启的对话文件在结构上完全一致，都出现在树视图中，都可以打开查阅。

## 三、数据流

### 3.1 正常流程（无异常）

```
用户消息
  │
  ▼
LLMChatService.sendMessageStream()
  │  ① 写入用户消息到 PA 对话文件
  │  ② 构建 LLM 消息（含系统指令 + 历史）
  │  ③ 调用 LLMService.streamWithTools(PERSONAL_ASSISTANT_TOOLS)
  │
  ▼
执行官 LLM 思考
  │
  ├─► 调用 read_memory → 从记忆文件读取历史
  ├─► 调用 list_chat_roles → 获取团队名单
  │
  ├─► 调用 delegate_to_role("研究员", "调研XXX")
  │     │
  │     ▼
  │   PersonalAssistantService.delegateToRole()
  │     │  ① createConversation(研究员.id) → 创建对话文件
  │     │  ② appendMessage(uri, 'user', task) → 写入任务
  │     │  ③ LLMService.streamWithTools([研究员系统提示 + task])
  │     │       └─► 研究员可调用 CHAT_TOOLS（搜索、创建笔记等）
  │     │  ④ appendMessage(uri, 'assistant', reply) → 写入回复
  │     │  ⑤ 刷新树视图
  │     │
  │     └─► 返回研究员的回复文本
  │
  ├─► 调用 evaluate_role("研究员", "success", "...") → 追加到记忆文件
  ├─► 调用 write_memory(新记忆内容) → 更新记忆文件
  │
  └─► 生成最终回复流式输出
        │
        ▼
      ④ appendMessage(PA对话, 'assistant', 最终回复)
      ⑤ fire onDidSendMessage → 更新 ChatHistoryPanel
```

### 3.2 文件产物

一次典型的 PA 交互后，文件系统变化：

```
issues/
├── {timestamp}-pa-role.md          # 执行官角色文件（仅创建一次）
├── {timestamp}-pa-memory.md        # 助手记忆文件（每次更新 body）
├── {timestamp}-pa-convo-1.md       # 与执行官的第 1 次对话
│     ├── User: 帮我研究XX
│     └── Assistant: 好的，我来...（最终汇报）
│
├── {timestamp}-researcher-role.md  # 研究员角色文件
└── {timestamp}-delegated-convo.md  # [助手委派] 研究XX...
      ├── User: [委派任务内容]
      └── Assistant: [研究员的完整回复]
```

## 四、状态机：`delegateToRole()`

每次委派经过以下状态节点，任何节点失败都有确定的处理策略：

```
┌─────────┐
│  START  │
└────┬────┘
     │
     ▼
┌─────────────────────────────────────────────┐
│ 查找角色（_findRole）                         │
│ 按名称或 ID 在所有聊天角色中搜索              │
└────┬──────────────────────────────┬──────────┘
     │ 找到                          │ 未找到
     ▼                              ▼
 继续                          返回错误文本
                               （无文件副作用）
     │
     ▼
┌─────────────────────────────────────────────┐
│ 创建对话文件（createConversation）            │
│ 标题：[助手委派] {任务前30字}                │
└────┬──────────────────────────────┬──────────┘
     │ 成功，convoUri 有值            │ 失败，convoUri = null
     ▼                              ▼
 继续                          继续（LLM 照常调用，
                               只是无持久化记录）
     │
     ▼
┌─────────────────────────────────────────────┐
│ 写入用户消息（appendMessageToConversation）  │
│ role='user', content=task                   │
└────┬────────────────────────────────────────┘
     │ （写入失败只记录 warning，不中断）
     ▼
┌─────────────────────────────────────────────┐
│ 调用目标角色 LLM（LLMService.streamWithTools）│
│ 传入：角色系统提示 + 任务消息 + CHAT_TOOLS   │
└────┬──────────────────────┬─────────────────┘
     │ 正常返回              │ 抛出异常
     ▼                      ▼
 reply 有内容？         ┌───────────────┐
     │                  │ signal.aborted│
     ├─ 是：写入助手消息  │    ？         │
     │   刷新视图        └───┬───────┬───┘
     │   返回回复文本        │是     │否
     │                      ▼       ▼
     └─ 否：写入          [已取消] [委派失败]
          '[未返回内容]'    标记     错误标记
          刷新视图         写入     写入对话文件
          返回提示         对话文件  刷新视图
                          刷新视图  返回错误文本
                          返回提示
     ▼
┌─────────┐
│   END   │
└─────────┘
```

**关键保证**：每个到达 LLM 调用步骤之后的结果，无论成功/失败/取消，都会在对话文件中写入对应的助手消息，不会留下「只有用户消息」的幽灵对话。

## 五、状态机：PA 主对话 `sendMessageStream()`

```
用户消息送达
     │
     ▼
① appendMessage(PA对话, 'user', userMessage)
     │ 失败 → 直接返回 null（不调用 LLM）
     ▼
② buildLLMMessages() ← 读取 PA 对话历史
     │
     ▼
③ LLMService.streamWithTools(PERSONAL_ASSISTANT_TOOLS)
     │
     ├─ 流式 chunk → onChunk() → ChatHistoryPanel 实时显示
     │
     ├─ 工具调用 → executePersonalAssistantTool()
     │    └─ delegate_to_role → delegateToRole()（见上节状态机）
     │
     ├─ signal.aborted → 返回 null（不写助手消息）
     │
     └─ 抛出异常 → showErrorMessage + 返回 null（不写助手消息）
          ▲
          │ 注意：此时 PA 对话文件中有用户消息但无助手消息
          │ 这是所有角色共有的行为（非 PA 特有问题）
          │ 用户可在下一轮对话继续，LLM 会看到这条孤立消息
     │
     ▼
④ appendMessage(PA对话, 'assistant', reply)
⑤ fire onDidSendMessage
```

## 六、状态隔离保证

| 操作 | 是否影响 LLMChatService 的 active 状态 | 说明 |
|------|--------------------------------------|------|
| delegateToRole | **否** | 直接操作文件 + LLMService，绕过 LLMChatService |
| create_chat_role | **否** | 直接调用 createChatRole()，无状态修改 |
| update_role_config | **否** | 直接修改角色文件的 frontmatter |
| evaluate_role | **否** | 追加内容到记忆文件 |
| read_memory / write_memory | **否** | 读写记忆文件，与对话状态无关 |

PA 在处理工具调用时，`LLMChatService._activeConversationUri` 始终指向 PA 自己的对话，不会被委派操作修改。工具调用完成后，PA 的 LLM 流式输出继续，最终写入 PA 对话文件。

---

## 七、并发安全性

`LLMService.streamWithTools` 的工具调用是**串行**的（当前 LLM 生成在工具执行期间暂停）。因此，即使 PA 在一次回复中多次调用 `delegate_to_role`，这些调用也是顺序执行的，不存在并发写入同一文件的风险。

## 八、记忆文件

### 8.1 结构

记忆文件是一个普通 issueMarkdown，frontmatter 标记为 `assistant_memory: true`：

```yaml
---
assistant_memory: true
assistant_role_id: "20241220-103045"   # 执行官角色文件的 ID
---
# 执行官记忆

## 用户背景
（PA 在对话中逐步积累的用户偏好、工作习惯）

## 历史任务
### 研究量子计算 — 2026-03-09 ✓ 良好
委派给研究员，耗时约 2 分钟，生成了 5 篇笔记。

## 角色绩效
### 研究员 — 2026-03-09 ✓ 良好
研究深度好，但有时过于冗长，建议在提示词中加"回复控制在500字以内"。

## 自我反思
对于代码相关任务，应先问用户技术栈，再委派给编程助手，避免角色猜测。
```

### 8.2 读写时机

| 时机 | 操作 | 调用方式 |
|------|------|---------|
| 每次对话开始 | `read_memory` | PA 的 system prompt 要求它主动调用 |
| 任务完成后 | `write_memory` | PA 的 system prompt 要求它更新 |
| 角色评估后 | `evaluate_role` | 工具内部自动追加到记忆文件的"角色绩效"章节 |

记忆文件的更新是**增量追加**式的（由 PA 的 LLM 自主决定写什么），不是结构化 JSON，而是自由格式 Markdown，这让 PA 能自主演化记忆的格式和粒度。

## 九、错误恢复策略

| 异常场景 | 文件状态 | 用户可感知 | 恢复建议 |
|---------|---------|-----------|---------|
| 委派时 LLM 报错 | 对话文件有用户消息 + `[委派失败]` 标记 | 错误文本返回给 PA，PA 告知用户 | PA 可在下一工具调用中重试 |
| 委派被用户取消 | 对话文件有用户消息 + `[已取消]` 标记 | PA 收到"已取消"文本 | 用户可重新发消息触发新一轮 |
| 角色 LLM 返回空 | 对话文件有用户消息 + `[未返回内容]` 标记 | PA 收到提示，可尝试换角色 | PA 可委派给其他角色 |
| PA 自身 LLM 报错 | PA 对话文件只有用户消息，无助手消息 | Toast 错误提示 | 直接重发消息即可（LLM 会看到历史） |
| createConversation 失败 | 无对话文件，LLM 仍然调用 | PA 正常运行，但无持久记录 | 检查磁盘空间/文件权限 |
| write_memory 失败 | 记忆未更新，其余正常 | 无感知 | 下次任务时 PA 会再次尝试 |

```

├── 👤 执行官（个人助手）           ← PersonalAssistantNode（顶部固定）
│   ├── 与执行官的对话 1             ← 刚创建时打开这个
│   └── 与执行官的对话 2
│
├── 👥 技术评审组                   ← 群组节点
│   └── 评审会议 1
│
├── 🔍 深度研究员                   ← 普通角色节点
```

委派产生的对话标题以 `[助手委派]` 前缀标识，与用户手动开启的对话一目了然。

## 十一、自我进化机制

PA 的「自我进化」通过以下工具组合实现：

```
任务完成后：

evaluate_role("研究员", "partial", "回复太长，建议限制字数")
  └─ 追加到记忆文件的"角色绩效"章节

update_role_config("研究员", "新的系统提示词（加了字数限制）")
  └─ 直接更新研究员的 issueMarkdown frontmatter 中的 chat_role_system_prompt

write_memory(新记忆)
  └─ 记录本次进化决策
```

下次 PA 再委派研究员时，研究员的行为就已经被优化了。

## 十二、模型配置

### 12.1 优先级（从高到低）

```
角色文件 chat_role_model_family
        ↓（未设置时）
VS Code 设置 issueManager.llm.modelFamily
        ↓（未设置时）
代码硬编码默认值：gpt-5-mini
        ↓（模型不存在时，依次 fallback）
gpt-4o → gpt-4.1 → 任意可用 Copilot 模型
```

### 12.2 可选模型

| 模型 | 定位 | 适用场景 |
|------|------|---------|
| `gpt-5-mini` | 快速轻量（**默认**） | 日常对话、简单委派、记忆读写 |
| `gpt-4o` | 均衡能力 | 复杂推理、多步骤任务 |
| `gpt-4.1` | 旗舰深度 | 长文本分析、深度研究、架构设计 |
| `gpt-4.1-mini` | 4.1 轻量版 | 速度与质量兼顾 |
| `o3-mini` | 推理专用 | 逻辑、数学、代码分析 |
| `claude-3.5-sonnet` | Anthropic 旗舰 | 代码生成、细致分析 |

### 12.3 配置方式

**全局默认**（影响所有未单独配置模型的角色）：
```jsonc
// .vscode/settings.json
{
  "issueManager.llm.modelFamily": "gpt-5-mini"
}
```

**角色级配置**（创建角色时选择，或直接编辑角色 `.md` 文件的 frontmatter）：
```yaml
---
chat_role: true
chat_role_name: 深度研究员
chat_role_model_family: gpt-4.1   # 该角色专用模型，覆盖全局默认
---
```

**PA 工具动态创建角色时指定模型**（`create_chat_role` 工具的 `modelFamily` 参数）：
```
create_chat_role(
  name="法律顾问",
  systemPrompt="...",
  modelFamily="gpt-4.1"   # 需要高精度推理，指定旗舰模型
)
```

## 十三、与普通角色的区别

| 维度 | 普通角色 | 个人助手 |
|------|---------|---------|
| 工具集 | CHAT_TOOLS（笔记 + 浏览器） | PERSONAL_ASSISTANT_TOOLS（全部 + 记忆 + 委派 + 角色管理） |
| 对话状态 | 无跨对话记忆 | 持久记忆文件，跨对话积累 |
| 角色关系 | 独立人格 | 用户代理，可指挥其他角色 |
| 树视图位置 | 普通列表项 | 顶部固定（PersonalAssistantNode） |
| 委派产生的对话 | 不适用 | 出现在被委派角色下，可独立查看 |
| 进化能力 | 无（需手动编辑文件） | 可通过工具自主优化自身和其他角色 |
| 默认模型 | gpt-5-mini（全局默认） | gpt-5-mini（可在角色文件中单独配置） |
